# 03 - 技术架构文档

> 本文档是项目的技术核心文档，主要面向程序开发（含 AI 辅助编码）。  
> 设计原则：**简洁、够用、易维护**。不引入复杂框架，保持原生 Unity 开发方式。

## 1. 架构设计原则

1. **Manager 单例模式**：核心系统用少量 Manager 管理，每个 Manager 职责清晰
2. **数据驱动**：关卡配置与代码逻辑分离，用 ScriptableObject 存储关卡数据
3. **简单状态机**：用枚举管理游戏状态，不引入复杂状态机框架
4. **不过度设计**：够用就好，不为"可能用到"的功能预留复杂接口

### 1.1 架构演进方向：Prefab 化（2026-04-24 新增）

> **背景**：当前 M1/M2 的所有 GameObject 和 UI 都是纯代码在运行时动态创建的（`SceneBootstrap` 一个脚本启动一切）。这种方式快速原型很高效，但存在以下问题：
> - 设计师/技术美术**无法在 Inspector 中手动调整**任何参数（间距、大小、颜色等）
> - **无法直接挂载特效组件**（粒子系统、动画控制器等）
> - 所有改动都必须修改代码，**非程序人员无法介入**

**改进目标**：逐步把关键游戏元素从代码动态创建改为 **Prefab + Inspector 可配置** 的形式。

**Prefab 化对象清单：**

| 对象 | Prefab 路径 | 状态 |
|------|------------|------|
| 棋盘格子 | `Prefabs/Board/Cell.prefab` | ✅ 已创建 |
| 棋盘预览 | `Prefabs/Board/Preview.prefab` | ✅ 已创建 |
| 候选区插槽 | `Prefabs/Board/CandidateSlot.prefab` | ✅ 已创建 |
| BoardManager | `Prefabs/Board/[BoardManager].prefab` | ✅ 已创建 |
| BlockSpawner | `Prefabs/Board/[BlockSpawner].prefab` | ✅ 已创建 |
| 方块单元格 | `Prefabs/Block/BlockCell.prefab` | ✅ 已创建 |
| 得分飘字 | `Prefabs/UI/FloatingScore.prefab` | ✅ 已创建 |
| GameOver 面板 | `Prefabs/UI/GameOverPanel.prefab` | ✅ 已创建 |
| 最高分显示 | `Prefabs/UI/HighScoreDisplay.prefab` | ✅ 已创建 |
| HUD 重启按钮 | `Prefabs/UI/HudRestartButton.prefab` | ✅ 已创建 |
| 当前分数显示 | `Prefabs/UI/ScoreDisplay.prefab` | ✅ 已创建 |
| 分数文本 | `Prefabs/UI/ScoreText.prefab` | ✅ 已创建 |

**Prefab 化后的好处：**
1. 可以直接在 Prefab 上**挂载粒子特效/动画组件**
2. 可以在 Inspector 中**调间距、大小、颜色**等参数
3. 可以调整**拖拽手感**（缩放系数、偏移量等）而不改代码
4. 设计师可以通过**修改 Prefab** 来微调表现，不依赖程序

**实施策略：**
- 代码中改为 `Instantiate(prefab)` 方式创建对象
- 通过 `[SerializeField]` 在 Inspector 中引用 Prefab
- 保留代码创建 fallback（Prefab 引用为 null 时走原有代码路径，确保不崩溃）
- `SceneBootstrap` 改为从场景中读取配置，而非全部硬编码

## 4. 核心 Manager 系统

### 4.1 系统总览

```
GameManager（游戏总管理）
├── BoardManager    - 棋盘逻辑（放置、消除）
├── LevelManager    - 关卡加载、目标判定
├── ScoreManager    - 计分
├── UIManager       - 界面切换与更新
├── AudioManager    - 音效播放
└── BlockSpawner    - 候选方块生成
```

### 4.2 各 Manager 职责

#### GameManager（游戏总管理器）

- 管理游戏全局状态（主菜单 / 游戏中 / 暂停 / 结算）
- 协调各 Manager 之间的通信
- 控制游戏流程（开始、暂停、结束）
- **生命周期**：全局单例，不销毁

#### BoardManager（棋盘管理器）

- 创建和维护棋盘数据（二维数组）
- 处理方块放置逻辑（判断是否可放、执行放置）
- 调用 MatchChecker 检测并执行消除
- **生命周期**：游戏场景中存在

#### LevelManager（关卡管理器）

- 加载关卡配置数据（ScriptableObject）
- 管理当前关卡的步数和目标
- 判定关卡是否通过
- 管理关卡进度存档
- **生命周期**：全局单例

#### ScoreManager（计分管理器）

- 从 `SceneBootstrap` 注入的 `ScoreConfig` 读取每格放置分、排分表、Combo 加成系数、奖励机会数等配置
- 若未注入配置，则兜底加载 `Resources/Configs/ScoreConfig`
- 计算放置得分、消除基础加分和 Combo 加成
- 管理 Combo 数与 Combo 奖励机会状态
- 提供当前分数、最高分和计分详情事件
- **生命周期**：游戏场景中存在

#### UIManager（UI 管理器）

- 管理所有 UI 面板的显示/隐藏
- 更新游戏中 HUD（分数、步数等）
- **生命周期**：全局单例

#### AudioManager（音效管理器）

- 播放音效（SFX）和背景音乐（BGM）
- 管理音量设置
- **生命周期**：全局单例，不销毁

### 4.3 Manager 之间的通信方式

采用**直接引用 + C# 事件**的简单方式：

```csharp
// 示例：方块放置后的事件链
// BoardManager 放置方块 → 触发事件
public static event Action<int> OnBlockPlaced;      // 参数：放置的格子数
public static event Action<int> OnLinesCleared;     // 参数：消除的行/列数

// ScoreManager 监听事件 → 计算分数
// UIManager 监听事件 → 更新显示
// AudioManager 监听事件 → 播放音效
// LevelManager 监听事件 → 判定是否过关
```

> 不使用消息总线、不使用复杂的事件系统，直接用 C# 原生的 `event` / `Action` 就够了。

## 5. 数据结构设计

### 5.1 棋盘数据

```csharp
// 棋盘用一个二维数组表示
bool[,] boardGrid;  // true = 已占用, false = 空

// 棋盘大小
int rows = 8;
int cols = 8;
```

### 5.2 方块形状数据

```csharp
// 方块形状用相对坐标数组表示
// 例如 L 形：
// ■
// ■■
// 坐标: (0,0), (1,0), (1,1)  （以左上角为原点）

[System.Serializable]
public class BlockShapeData
{
    public string shapeName;
    public Vector2Int[] cells;  // 组成方块的格子相对坐标
    public int weight;          // 生成权重（出现概率）
}
```

### 5.3 关卡配置数据（ScriptableObject）

```csharp
[CreateAssetMenu(fileName = "LevelData", menuName = "Game/LevelData")]
public class LevelData : ScriptableObject
{
    public int levelId;
    public int targetScore;
    public int maxSteps;
    public int boardRows;
    public int boardCols;
    public int star2Score;      // 2 星分数线
    public int star3Score;      // 3 星分数线
    // public Vector2Int[] preplacedBlocks;  // 预置方块（待扩展）
}
```

## 6. 游戏状态管理

```csharp
public enum GameState
{
    MainMenu,       // 主菜单
    LevelSelect,    // 关卡选择
    Playing,        // 游戏进行中
    Paused,         // 暂停
    GameOver,       // 游戏结束（失败）
    LevelComplete   // 关卡完成（过关）
}
```

状态切换由 `GameManager` 统一管理，切换时通知 `UIManager` 更新界面。

## 7. 场景规划

| 场景 | 用途 | 说明 |
|------|------|------|
| MainMenu | 主菜单 + 关卡选择 | 可合并为一个场景 |

## 8. 代码规范

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase | `BoardManager` |
| 公有方法 | PascalCase | `PlaceBlock()` |
| 私有方法 | PascalCase | `CheckMatch()` |
| 公有变量 | camelCase | `currentScore` |
| 私有变量 | _camelCase | `_boardGrid` |
| 常量 | UPPER_SNAKE | `MAX_BOARD_SIZE` |
| 枚举值 | PascalCase | `GameState.Playing` |

### 其他约定

- 每个脚本文件只放一个类
- `MonoBehaviour` 脚本用 `[SerializeField]` 暴露私有字段，避免用 `public`
- 优先用 `[SerializeField]` + Inspector 赋值，减少 `Find` / `GetComponent` 调用
- 注释用中文，方便团队阅读

## 10. 关卡编辑工具

采用 **ScriptableObject + Unity Inspector** 方案，设计师直接在 Inspector 中配置关卡参数。

LevelData 数据结构见 §5.3，关卡文件放在 `Assets/ScriptableObjects/Levels/`。

| 难度 | 步数 | 目标分 | 说明 |
|------|------|--------|------|
| 简单 | 20~30 | 100~300 | 前 5 关，让玩家熟悉操作 |
| 中等 | 15~25 | 300~600 | 需要有意识地制造消除 |
| 困难 | 10~20 | 500~1000 | 需要多行同时消除才能达标 |

后续扩展（M3）：预置方块、限制方块池、可视化关卡编辑器。
