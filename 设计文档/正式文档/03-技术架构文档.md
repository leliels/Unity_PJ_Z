# 03 - 技术架构文档

> 本文档是项目的技术核心文档，主要面向程序开发（含 AI 辅助编码）。  
> 设计原则：**简洁、够用、易维护**。不引入复杂框架，保持原生 Unity 开发方式。

## 1. 架构设计原则

1. **Manager 单例模式**：核心系统用少量 Manager 管理，每个 Manager 职责清晰
2. **数据驱动**：模式、计分、方块形状、音效等配置与代码逻辑分离，优先用 ScriptableObject / Prefab Inspector 管理
3. **简单状态机**：用枚举管理游戏状态，不引入复杂状态机框架
4. **不过度设计**：够用就好，不为"可能用到"的功能预留复杂接口

### 1.1 架构演进方向：Prefab 化（2026-04-24 新增）

> **背景**：当前 M1/M2 的所有 GameObject 和 UI 都是纯代码在运行时动态创建的（`SceneBootstrap` 一个脚本启动一切）。这种方式快速原型很高效，但存在以下问题：
> - 设计师/技术美术**无法在 Inspector 中手动调整**任何参数（间距、大小、颜色等）
> - **无法直接挂载特效组件**（粒子系统、动画控制器等）
> - 所有改动都必须修改代码，**非程序人员无法介入**

**改进目标**：逐步把关键游戏元素从代码动态创建改为 **Prefab + Inspector 可配置** 的形式。

**Prefab 化对象清单：**

| 对象 | Prefab 路径 | 状态 |
|------|------------|------|
| 棋盘格子 | `Prefabs/Board/Cell.prefab` | ✅ 已创建 |
| 棋盘预览 | `Prefabs/Board/Preview.prefab` | ✅ 已创建 |
| 候选区插槽 | `Prefabs/Board/CandidateSlot.prefab` | ✅ 已创建 |
| BoardManager | `Prefabs/Board/[BoardManager].prefab` | ✅ 已创建 |
| BlockSpawner | `Prefabs/Board/[BlockSpawner].prefab` | ✅ 已创建 |
| 方块单元格 | `Prefabs/Block/BlockCell.prefab` | ✅ 已创建 |
| 得分飘字 | `Prefabs/UI/FloatingScore.prefab` | ✅ 已创建 |
| GameOver 面板 | `Prefabs/UI/GameOverPanel.prefab` | ✅ 已创建 |
| 最高分显示 | `Prefabs/UI/HighScoreDisplay.prefab` | ✅ 已创建 |
| HUD 重启按钮 | `Prefabs/UI/HudRestartButton.prefab` | ✅ 已创建 |
| 当前分数显示 | `Prefabs/UI/ScoreDisplay.prefab` | ✅ 已创建 |
| 分数文本 | `Prefabs/UI/ScoreText.prefab` | ✅ 已创建 |

**Prefab 化后的好处：**
1. 可以直接在 Prefab 上**挂载粒子特效/动画组件**
2. 可以在 Inspector 中**调间距、大小、颜色**等参数
3. 可以调整**拖拽手感**（缩放系数、偏移量等）而不改代码
4. 设计师可以通过**修改 Prefab** 来微调表现，不依赖程序

**实施策略：**
- 代码中改为 `Instantiate(prefab)` 方式创建对象
- 通过 `[SerializeField]` 在 Inspector 中引用 Prefab
- 保留代码创建 fallback（Prefab 引用为 null 时走原有代码路径，确保不崩溃）
- `SceneBootstrap` 改为从场景中读取配置，而非全部硬编码

## 4. 核心 Manager 系统

### 4.1 系统总览

```text
Title 场景
├── TitleUI             - 模式入口、设置入口
├── ModeManager         - 当前模式与模式配置
├── SaveManager         - 本地设置、分数、游玩记录
├── AudioManager        - 音乐/SFX 总控
└── FeedbackManager     - 震动与 PC 抖动反馈

传统模式游戏场景（当前 Boot）
├── SceneBootstrap      - 创建/连接游戏场景对象
├── GameManager         - 游戏流程与状态
├── BoardManager        - 棋盘逻辑（放置、消除、GameOver）
├── BlockSpawner        - 3 个候选方块生成
├── ScoreManager        - 计分与最高分事件
├── GameUI              - 游戏 HUD / GameOver
└── GameplayAudioBinder - 监听玩法事件并播放音效/反馈
```

### 4.2 各 Manager 职责

#### GameManager（游戏流程管理）

- 管理游戏中状态：开始、暂停、继续、GameOver、重新开始。
- 不保存具体 AudioClip，不直接管理每个按钮/方块的声音。
- 可以发出高层事件，供 UI、存档、音效绑定组件监听。
- **生命周期**：游戏场景中存在。

#### ModeManager（模式管理）

- 读取模式配置，生成 Title 上的模式入口。
- 管理当前模式 ID、启动场景、占位状态。
- 传统模式进入当前游戏场景；冒险模式本阶段只显示占位提示。

#### SaveManager（本地数据）

- 统一读写 `PlayerPrefs`。
- 保存全局设置、各模式最高分、游玩次数、累计分、最近游玩时间和有限数量的单局记录。
- 提供清除用户数据接口，供 Title 设置调用。

#### AudioManager（音效总控）

- 独立运行时服务，不从属于 `GameManager`。
- 管理 BGM、SFX、全局开关、音量、AudioSource 池和 `AudioLibrary`。
- 对外提供统一播放接口，具体触发由各音效触发脚本负责。

#### FeedbackManager（震动/抖动反馈）

- 移动端调用系统震动。
- PC/Editor 下执行短促棋盘根节点抖动，方便开发期验证震动逻辑。
- 抖动只移动根 Transform，避免逐格位移导致错位。

#### UIManager / TitleUI / GameUI

- `TitleUI`：Title 模式入口、全局设置弹窗、清除数据确认。
- `GameUI`：游戏内 HUD、设置/暂停、GameOver 结算。
- `UIManager`：公共弹窗切换、面板显示/隐藏。

### 4.3 通信方式

继续采用**直接引用 + C# 事件**，不引入消息总线。

```csharp
// 示例：玩法事件链
BoardManager.OnBlockPlaced      → ScoreManager / GameplayAudioBinder
BoardManager.OnLinesCleared     → ScoreManager / FloatingScoreManager / GameplayAudioBinder
ScoreManager.OnScoreChanged     → GameUI / SaveManager
GameManager.OnGameStateChanged  → GameUI / SaveManager / GameplayAudioBinder
```

音效逻辑不要散落进核心玩法代码：玩法代码只发事件，`GameplayAudioBinder` 负责把事件映射到 `AudioCue` 和反馈。

## 5. 数据结构设计

### 5.1 棋盘数据

```csharp
// 棋盘用一个二维数组表示
bool[,] boardGrid;  // true = 已占用, false = 空

// 棋盘大小
int rows = 8;
int cols = 8;
```

### 5.2 方块形状数据

方块形状采用 **ScriptableObject 数据库 + EditorWindow 配置工具** 管理。具体操作见 [操作指南/方块形状配置工具使用指南](操作指南/方块形状配置工具使用指南.md)。

运行时核心结构：

```csharp
[System.Serializable]
public class BlockData
{
    public string ShapeName;
    public Vector2Int[] Cells;  // 相对坐标（以左下角为原点）
    public int Weight;          // 生成权重（出现概率）
}
```

配置资产结构：

```csharp
public sealed class BlockShapeDatabase : ScriptableObject
{
    string databaseId;
    string displayName;
    int editorGridColumns;
    int editorGridRows;
    List<BlockShapeDefinition> shapes;
}

[System.Serializable]
public sealed class BlockShapeDefinition
{
    string id;
    string displayName;
    bool enabled;
    int weight;
    List<Vector2Int> cells;
    string note;
}
```

`BlockSpawner` 优先从当前配置的 `BlockShapeDatabase` 中按权重随机出块；若配置缺失或无有效形状，则回退到 `BlockData.GetAllShapes()` 内置形状表。

### 5.3 模式配置数据

```csharp
public sealed class GameModeConfig : ScriptableObject
{
    string modeId;          // traditional / adventure
    string displayName;     // Title 按钮文案
    bool enabled;           // 是否可进入
    bool placeholder;       // 是否只显示占位提示
    string sceneName;       // 启动场景
    int sortOrder;          // 显示顺序
    string description;     // 简短说明
}
```

### 5.4 本地存档数据

M3 仍使用 `PlayerPrefs`，但不要把 key 散落到各个系统。建议由 `SaveManager` 统一维护：

| 数据 | 示例 Key | 说明 |
|------|----------|------|
| 设置 | `BlockPuzzle_Settings` | 音乐/音效/震动开关与音量 |
| 模式最高分 | `BlockPuzzle_Mode_traditional_HighScore` | 按模式保存最高分 |
| 模式摘要 | `BlockPuzzle_Mode_traditional_Summary` | 游玩次数、累计分、最近时间 |
| 游玩记录 | `BlockPuzzle_PlayRecords` | 有数量上限的 JSON 记录列表 |

## 6. 游戏状态管理

M3 先保持简单状态，不做复杂状态机：

```csharp
public enum GameState
{
    Title,          // Title 启动界面
    Playing,        // 游戏进行中
    Paused,         // 游戏内设置/暂停
    GameOver        // 当前局结束
}
```

模式状态由 `ModeManager` 管理；游戏中状态由 `GameManager` 管理。二者通过明确接口通信，避免互相承担对方职责。

## 7. 场景规划

| 场景 | 用途 | 说明 |
|------|------|------|
| Title | 启动界面 | 模式入口、全局设置、清除用户数据 |
| Boot / TraditionalGame | 传统模式游戏场景 | 当前已有玩法入口；后续可复制或重命名为更明确的传统模式场景 |

> M3 先保持 2 个核心场景：`Title` + 传统模式游戏场景。冒险模式本阶段不创建正式玩法场景，只显示占位入口。

## 8. 代码规范

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase | `BoardManager` |
| 公有方法 | PascalCase | `PlaceBlock()` |
| 私有方法 | PascalCase | `CheckMatch()` |
| 公有变量 | camelCase | `currentScore` |
| 私有变量 | _camelCase | `_boardGrid` |
| 常量 | UPPER_SNAKE | `MAX_BOARD_SIZE` |
| 枚举值 | PascalCase | `GameState.Playing` |

### 其他约定

- 每个脚本文件只放一个类
- `MonoBehaviour` 脚本用 `[SerializeField]` 暴露私有字段，避免用 `public`
- 优先用 `[SerializeField]` + Inspector 赋值，减少 `Find` / `GetComponent` 调用
- 注释用中文，方便团队阅读

### 12.3 开发规则

1. **新增编辑器工具**时，必须放在 `BlockPuzzle/` 菜单下，不得使用 `Tools/` 或其他顶层菜单
2. 如果是人（策划/美术/程序）日常要用的工具 → 直接放 `BlockPuzzle/` 一级
3. 如果是 AI 开发期间一次性使用的生成脚本 → 放 `BlockPuzzle/AI 工具/`
4. MenuItem priority 建议：日常工具 10~99，AI 工具 200+
