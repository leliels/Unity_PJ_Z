# 09 - 游戏内术语表

> 本文档用于统一团队沟通中的游戏内对象、规则、界面、资源与代码术语。  
> 原则：**一个术语优先在一个主模块中定义**；如果它也出现在其他模块，只做引用，不重复展开。  
> 当前版本：v0.1 / 2026-05-01

## 1. 使用规则

### 1.1 表格字段说明

| 字段 | 说明 |
|---|---|
| 标准术语 | 团队沟通时优先使用的中文叫法 |
| 英文/代码名 | 代码、Prefab、资源或配置中的对应命名 |
| 含义 | 这个术语具体指什么，避免歧义 |
| 不推荐叫法/易混叫法 | 可以听懂但不建议继续扩散的叫法 |
| 主归属 | 该术语的主定义模块 |

### 1.2 命名原则

1. **“棋盘格”只指棋盘上的格子**；组成方块的小格统一叫 **“方块单格”**。
2. **“方块”默认指玩家拖拽的整块形状组合**，不是单个小格。
3. **“候选方块”指底部 3 个可选择方块之一**；“候选区”指承载这 3 个方块的区域。
4. **“排”在计分中是行/列的统称**；“消除排数” = 本次消除的行数 + 列数。
5. **“Combo CD”保留英文缩写**，中文可说“Combo 冷却值”或“续连冷却值”。
6. 美术文件名、Prefab 名、脚本类名保持英文/代码名；中文沟通优先用本文档的标准术语。

| 本次得分 | `TotalScore` | `CellScore + ClearComboScore` | 单次分数 | 计分 |

当前正式计分公式：

```text
本次得分 = C × D × R + B × M × L × (L + 1)
当前分 A = 当前分 A + 本次得分
```

## 14. 已发现的不统一/需后续同步项

| 问题 | 当前情况 | 建议统一为 |
|---|---|---|
| `Title` / `Boot` / `Game` 场景叫法混用 | 当前实际运行是 `Boot.unity`，M3 规划新增 `Title` 启动界面 | 现阶段说“Title 启动界面”和“传统模式游戏场景（当前 Boot）”；后续实现后再统一场景名 |
| 8×8、竖屏、不可旋转在旧文档中仍带“待确认” | `02/04/05` 部分段落仍保留早期待确认说明 | 本术语表按已确认结果：8×8、竖屏 1080×1920、玩家不可旋转 |
| “方块”“格子”“小方块”容易混用 | 口头交流中经常把棋盘格和方块单格都叫小方块 | 棋盘上的叫“棋盘格”；方块内部组成单位叫“方块单格” |
| “连消”与“Combo”容易混用 | 设计文档中“同时消除”曾写“连消奖励”，新版计分使用 Combo CD | 同时消多行/列叫“同时消除”；跨回合续连状态叫“Combo” |
| `Assets/Configs/BlockShapes` 与 `Assets/Resources/Configs` 路径存在差异 | 文档写过默认数据库在 `Assets/Configs/BlockShapes`，当前也存在 `Resources/Configs/DefaultBlockShapeDatabase.asset` | 后续确认运行时实际绑定路径后统一文档；术语上统一叫“默认方块形状数据库” |
| `HighScoreDisplay 1.prefab` 与 `HighScoreDisplay.prefab` 并存 | 当前 Prefab 目录存在重复/临时文件痕迹 | 标准术语只使用 `HighScoreDisplay.prefab`，临时副本后续清理 |
| `CandidateBoard` 与 `CandidateSlot` 都可能被叫“候选框” | 一个是底板装饰，一个是方块容器 | 底板叫“候选区底板”；容器叫“候选槽位” |

## 15. 推荐口头表达示例

| 场景 | 推荐说法 | 不推荐说法 |
|---|---|---|
| 描述棋盘上已有内容 | “这个位置已经是占用格了。” | “这个小方块有东西。” |
| 描述拖动的对象 | “把这个候选方块拖到棋盘左下角。” | “把这个格子拖上去。” |

| 描述消除 | “这次同时消除 2 排，`L=2`。” | “这次连消 2 个。” |
| 描述候选区刷新 | “3 个候选方块都用完后刷新一组。” | “下面的格子用完刷新。” |
| 描述预览 | “有效放置时显示放置预览；如果能消除，再显示消除预览高亮。” | “出现白影/亮一下。” |
| 描述计分 | “消除时先算格子得分项，再算消除/Combo 得分项。” | “消了就随便加分。” |
